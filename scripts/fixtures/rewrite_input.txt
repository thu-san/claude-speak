I found the issue — the test was racy because `setupTestDB()` and the `time.Now()` call inside `TestTokenExpiry` were being evaluated in the wrong order. On a slow CI runner the token was already stale by the time the assertion ran.

Here's the fix:

```go
func TestTokenExpiry(t *testing.T) {
    db := setupTestDB(t)
    defer db.Close()

    issuedAt := time.Now()
    token, err := auth.IssueToken(db, userID, 5*time.Minute)
    require.NoError(t, err)

    // Freeze time at issuance instead of re-reading the clock — the
    // DB setup above can take 300ms+ on a cold start.
    clock := fakeclock.NewAt(issuedAt)
    svc := auth.NewService(db, clock)

    clock.Advance(4 * time.Minute)
    require.True(t, svc.IsValid(token), "token should still be valid at 4min")

    clock.Advance(2 * time.Minute)
    require.False(t, svc.IsValid(token), "token should be expired after 5min total")
}
```

Three things I changed:

1. Captured `time.Now()` once at issuance into `issuedAt` so the test isn't sensitive to how long `setupTestDB` takes.
2. Swapped the real clock out for `fakeclock` which we already use in the billing tests — no new deps.
3. Tightened the second `Advance` so the sum is exactly 6 minutes past issuance. Before it was 5m1s which was also fine in theory but meant any clock-drift bug wouldn't show up.

I ran it 100 times locally with `go test -count=100 -race ./auth/...` and it's green. I'd still recommend leaving `-race` on in CI for this package since auth touches goroutines in the session cache.
